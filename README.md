# 🚀 EVE Trade Bot

Автоматизированная система мониторинга рынка EVE Online через официальный публичный API (ESI): сканирует 5 крупнейших торговых хабов, считает межрегиональный арбитраж с учётом глубины стакана (не просто "лучшая цена", а средневзвешенная цена реального объёма), налогов и брокерской комиссии, и присылает в Telegram сводку по найденным возможностям. Отдельный workflow раз в сутки пересматривает список отслеживаемых товаров с помощью LLM — на основе накопленной статистики, а не вслепую.

## Архитектура: 3 workflow + Postgres

```
WF1 «Market Scan» (каждые 30 мин)
Schedule → Load Watchlist → Fetch Market Books (5 хабов) ─┬→ Prepare Snapshots → Save Snapshots
                                                            └→ Arbitrage Calc → Filter → Log & Dedup
                                                               → Resolve Station Names → AI Risk Check
                                                               → Compose Message (сводка) → Telegram

WF2 «History Sync» (ежедневно, 14:40)
Schedule → Load Watchlist → Fetch History (ESI /markets/history/, 5 хабов) → Upsert History

WF3 «Watchlist Review» (ежедневно, 22:00)
Schedule → Load Aggregates (тренды/волатильность из истории) → Load Candidates
        → AI Review (DeepSeek) → Validate & Apply Rules (лимиты и лок — в коде, не в промпте)
        → Build Report → Telegram
        → Log Changes → If (add/remove) → Apply Add / Apply Remove
```

### WF1 — сбор данных и расчёт арбитража

- **Load Watchlist** — список отслеживаемых товаров + средний дневной объём по каждому региону (для целевого объёма сделки)
- **Fetch Market Books** — для каждого товара тянет ордера сразу в 5 хабах (Jita, Amarr, Dodixie, Rens, Hek) и **обходит стакан ордеров** (не берёт только верхнюю строчку): считает средневзвешенную цену (VWAP) закупки/продажи целевого объёма — 5% от реального дневного оборота, а не фиксированное число
- **Arbitrage Calc** — для каждой пары хабов считает **две стратегии**: мгновенный флип (продажа в существующий buy-ордер назначения, без брокерской комиссии) и размещение своего sell-ордера (дороже, но с ограничением по дневному обороту назначения). Каждый прибыльный маршрут — отдельная запись, не только "лучший" на товар
- **Filter + Log & Dedup Opportunity** — отсекает то, что ниже порога прибыли, и не повторяет одно и то же уведомление чаще раза в 6 часов (проверка на стороне БД)
- **AI Risk Check** — LLM (DeepSeek) оценивает каждую возможность на признаки аномалии (не путает консервативный расчётный объём с "малым объёмом = подозрительно")
- **Compose Message** — не шлёт сообщение на каждую возможность отдельно, а собирает **одну сводку**: топ-5 по прибыли + суммарная потенциальная прибыль и требуемый капитал по всем найденным

### WF2 — история цен

Раз в сутки тянет дневную историю (`/markets/{region}/history/`) по всем товарам watchlist во всех 5 регионах — при добавлении нового товара история сразу доступна за ~90 дней (ESI хранит её сама), ждать накопления не нужно.

### WF3 — ротация watchlist

AI получает агрегаты по каждому товару (тренд 7/30 дней, волатильность, число найденных возможностей за 30 дней) и список кандидатов на замену — и предлагает, что убрать и что добавить. Модель **только предлагает**; жёсткие правила — в коде, а не в промпте:
- максимум 3 удаления и 3 добавления за один прогон
- новые товары получают лок на 7 дней (нельзя удалить, пока не накопится статистика)
- удалённые товары не возвращаются 14 дней (защита от чехарды)
- добавлять можно только `type_id` из заранее подготовленного пула кандидатов (никогда не по названию из головы — источник постоянных галлюцинаций у маленьких моделей)

## База данных

PostgreSQL, 6 таблиц: `watchlist` (что сканируем, с локами и источником изменения), `candidate_pool` (откуда AI выбирает замены), `price_snapshots` (внутридневная история), `daily_history` (дневная история из ESI, для трендов), `opportunities` (журнал найденного, с дедупликацией по 6-часовому окну), `watchlist_changes` (аудит решений AI). Полная схема — `db-schema.sql`.

## Экономия токенов AI

- Модель никогда не видит сырые ордера — только уже посчитанные кодом агрегаты (сотни токенов вместо десятков тысяч)
- AI Risk Check вызывается только для возможностей, прошедших фильтр и дедупликацию — не на каждый прогон
- WF3 работает раз в сутки, а не на каждом сканировании
- В промпт закладывается явный контекст ("объём — уже консервативная оценка, не повод для тревоги"), чтобы не тратить лишние вызовы и токены на переспрашивание

## Технологии

- **n8n** (self-hosted, Docker) — оркестрация всех трёх workflow
- **PostgreSQL** (self-hosted, Docker) — состояние watchlist, история цен, журнал возможностей
- **EVE Swagger Interface (ESI)** — официальный публичный API EVE Online (ордера, история цен, резолвинг названий)
- **DeepSeek API** — LLM для риск-анализа возможностей и ревью watchlist (дёшево, доступно из РФ)
- **JavaScript** (n8n Code nodes) — вся расчётная логика: обход стакана, VWAP, арбитраж, валидация решений AI
- **Telegram Bot API** — доставка сводок

## Как запустить

### 1. Поднять n8n и Postgres локально

```bash
docker volume create n8n_data
docker run -d --name n8n --restart unless-stopped -p 5678:5678 \
  -v n8n_data:/home/node/.n8n \
  -e GENERIC_TIMEZONE="Europe/Moscow" -e TZ="Europe/Moscow" \
  docker.n8n.io/n8nio/n8n

docker volume create trade_pg_data
docker run -d --name trade_postgres --restart unless-stopped -p 5434:5432 \
  -v trade_pg_data:/var/lib/postgresql/data \
  -e POSTGRES_USER=trade -e POSTGRES_PASSWORD=<свой пароль> -e POSTGRES_DB=evetrade \
  postgres:16
```

Откройте `http://localhost:5678`, создайте аккаунт. Примените схему:
```bash
docker cp db-schema.sql trade_postgres:/schema.sql
docker exec trade_postgres psql -U trade -d evetrade -f /schema.sql
```

### 2. Импортировать workflow

**Import from File** для каждого: `wf1-market-scan.json`, `wf2-history-sync.json`, `wf3-watchlist-review.json`.

### 3. Настроить credentials

- **Postgres**: host `host.docker.internal`, port `5434`, database `evetrade`, user/password — как задали при запуске контейнера
- **DeepSeek**: API-ключ с [platform.deepseek.com](https://platform.deepseek.com) (либо через нативную ноду, либо как OpenAI-совместимый credential с Base URL `https://api.deepseek.com`, модель `deepseek-chat`)
- **Telegram**: бот через [@BotFather](https://t.me/BotFather), Chat ID через [@userinfobot](https://t.me/userinfobot) — во всех трёх workflow замените плейсхолдеры `YOUR_TELEGRAM_CHAT_ID` и выберите credential

Во всех Postgres-нодах замените credential на созданный выше.

### 4. Наполнить candidate_pool и watchlist

`candidate_pool` — список товаров-кандидатов, резолвится через ESI `/universe/ids/` по названиям (см. пример в `db-schema.sql`). `watchlist` — стартовый набор для сканирования; дальше WF3 сам предлагает ротацию.

### 5. Активировать

Publish все три workflow. Первый запуск WF2 — вручную (бэкфилл 90 дней истории), дальше всё работает по расписанию само.

## Эксперимент: разговорный AI-агент (`ai-agent-workflow.json`)

Отдельный workflow, где вместо расписания — чат-бот на локальной LLM (Ollama, `llama3.1:8b`, self-hosted, без внешних API и оплаты), отвечающий на вопросы вроде "какая цена на Rifter в Jita?", сам решая, какие инструменты вызвать.

### Архитектура

```
When chat message received → AI Agent (Ollama Chat Model: llama3.1:8b)
                                 ├─ Tool: resolve_item_name   (POST /universe/ids/, находит type_id по названию)
                                 └─ Tool: get_best_sell_price (Code Tool: запрос ордеров + вычисление минимальной цены продажи)
```

### Что было исправлено в процессе отладки

- Неверный формат тела запроса (JS-объект вместо строки с валидным JSON)
- Захардкоженный `region_id` вместо параметра, который пыталась (и не должна была) угадывать модель
- Галлюцинация финального ответа — модель придумывала цену, даже если оба вызова инструментов падали с ошибкой; решено явным запретом в system prompt
- Неоднозначность ESI `/universe/ids/`: у предмета и у персонажа-игрока может совпадать имя, и оба попадают в ответ — понадобилось явно указать модели брать `type_id` только из `inventory_types[0].id`
- Отсутствующая/дефолтная Input Schema у Code Tool ноды (n8n по умолчанию подставляет заглушку `some_input`, из-за которой параметр `type_id` не доходил до кода)

### Найденное ограничение

`llama3.1:8b` (8B параметров, локально через Ollama) стабильно резолвит `type_id` предмета верно, но не всегда надёжно **переносит это число в следующий вызов инструмента** — иногда подставляет несуществующий ID вместо результата предыдущего шага. Это ограничение модели такого размера в multi-step tool chaining, а не баг конфигурации. В основной торговой системе (WF1-WF3) эта проблема обойдена архитектурно: AI никогда не передаёт числа между вызовами инструментов, вся оркестрация — в коде, модель только оценивает уже готовые данные.

## Возможные доработки

- Учёт реальных навыков персонажа (Accounting/Broker Relations) через ESI OAuth вместо усреднённых ставок налога
- Учёт грузоподъёмности и стоимости логистики между станциями (сейчас арбитраж не учитывает физическую перевозку между хабами)
- Расширение watchlist до сотен позиций (`candidate_pool` уже подготовлен, WF3 постепенно вводит новые товары через квоту ротации)
- Дашборд с историей найденных возможностей и статистикой по watchlist вместо только Telegram-уведомлений
